# Condor 2 Landscape File Formats - Phase 1 Reference

This document is a concise technical reference for the files and directory layout required to build a Condor 2 landscape, based on the installed sample `C:/Condor2/Landscapes/Slovenia2/`, the Condor Landscape Toolkit 2.7 (`D:/Repos/condor-landscape/tools/CLT2.7/`), and the *Condor Landscape Guide ENGLISH rev2*.

It covers the files needed for **Phase 1** (base terrain, airports, turnpoints, base textures/forests, and object placement) for the North Macedonia / Skopje landscape.

---

## 1. Coordinate system and landscape grid

Condor 2 uses a flat **UTM (WGS 84)** projection. A landscape is a rectangular matrix of **tiles** and **patches**.

| Unit | Size | Notes |
|------|------|-------|
| **Patch** | 5.76 x 5.76 km | Basic physical terrain unit. Heightmap files `.tr3` and forest files `.for` exist per patch. |
| **Tile** | 23.04 x 23.04 km | 4 x 4 patches. Texture files `.dds` exist per tile. |
| Grid origin | Bottom-right (south-east) | Tile `0000` is the bottom-right tile. Coordinates are `CCRR` (column, row), zero-padded to 4 digits. |

Patch coordinates use the same 4-digit `CCRR` scheme; a patch belongs to the tile whose index is `patch.CC // 4` and `patch.RR // 4`.

**Slovenia2 example:**

- `.trn` overview: `2560 x 1792` pixels at 90 m -> 10 x 7 tiles (230.4 x 161.28 km).
- Patch grid: `40 x 28 = 1120` patches -> `HeightMaps/` contains `h0000.tr3` ... `h3927.tr3` and `ForestMaps/` contains `0000.for` ... `3927.for`.
- Texture grid: `Textures/t0000.dds` ... `t3927.dds`, one per tile.

**North Macedonia / Skopje Phase 1 DEM:**

- Source: `sources/dem/macedonia_skopje_dem_utm30m.bil` / `.raw`
- Header: `2304 x 2304` pixels, 30 m resolution -> 69.12 x 69.12 km -> **3 x 3 tiles**, **12 x 12 patches = 144 patches**.

## 2. Landscape directory layout

A complete landscape lives under `C:/Condor2/Landscapes/<LandscapeName>/` and has this structure:

```text
<LandscapeName>/
├── <LandscapeName>.apt       # airports (binary)
├── <LandscapeName>.bmp       # PDA / flight-planner map (32-bit BMP)
├── <LandscapeName>.cup       # turnpoints / waypoints (SeeYou CSV)
├── <LandscapeName>.fha       # forest hash (text)
├── <LandscapeName>.ini       # landscape options (text)
├── <LandscapeName>.obj       # 3D object placements (binary)
├── <LandscapeName>.tdm       # thermal/albedo map (binary)
├── <LandscapeName>.tha       # terrain hash (text)
├── <LandscapeName>.trn       # source heightmap overview (binary)
│
├── Airports/                 # airport models and textures
│   ├── <ICAO>G.c3d / <ICAO>O.c3d   # ground / object models
│   └── <ICAO>/               # per-airport DDS textures
├── ForestMaps/
│   └── CCRR.for              # 512x512 forest mask per patch
├── HeightMaps/
│   └── hCCRR.tr3             # 193x193 height samples per patch
├── Images/
│   └── 0.jpg ... 6.jpg       # splash screen images
├── Textures/
│   ├── tCCRR.dds             # ground texture per tile
│   └── empty.dds             # fallback empty texture
├── World/
│   ├── Objects/*.c3d         # shared landscape objects
│   └── Textures/*.dds        # object textures
└── Working/                  # generated by Landscape Editor (optional at rest)
    ├── Terragen/Textures/*.bmp
    ├── Terragen/ForestMaps/*.bmp
    ├── ForestMaps/*.bmp
    ├── Textures/*.bmp
    └── ThermalMap.bmp
```

## 3. File format details

### 3.1 `<LandscapeName>.trn` - source heightmap overview

**Purpose:** Master elevation overview used by the Landscape Editor; source from which patch `.tr3` files are derived.

**Format:** binary.

- **Header - 36 bytes**
  - `width` (int32), `height` (int32) - overview dimensions in pixels.
  - 3 floats - rotation/tilt angles (usually `90.0, -90.0, 90.0`).
  - `easting`, `northing` (float32) - UTM coordinates of the **bottom-right (origin)** corner.
  - `utm_zone` (uint16), padding (uint16) - e.g. `33`.
  - `utm_hemisphere` (uint16, `'N'=78`), padding (uint16).
- **Data:** `width x height` unsigned 16-bit little-endian elevation values in **meters**.

**Slovenia2 example:**

- Size: `2560 x 1792` pixels.
- Header easting/northing: `616140.0 / 5040540.0`, UTM zone `33 N`.
- File size: `36 + 2560*1792*2 = 9 175 076` bytes.

**Created/edited by:**

- `RawToTrn.exe` from a GDAL raw/BIL/GeoTIFF heightmap.
- Condor_Tiles / `Condor_Tiles.exe` can also generate `.trn` from SRTM data.
- Landscape Editor reads it when opening the landscape.

---

### 3.2 `HeightMaps/hCCRR.tr3` - per-patch heightmap

**Purpose:** In-game terrain mesh; one file per patch.

**Format:** binary, no header.

- `193 x 193` unsigned 16-bit little-endian elevation values in **meters**.
- File size: `193*193*2 = 74 498` bytes.
- 193 samples = 192 intervals; at 30 m spacing this covers 5.76 km, matching one patch.

**Slovenia2 example:**

- `1120` files (`h0000.tr3` ... `h3927.tr3`).
- Value range observed: `0 m` to `2849 m` (Mt. Triglav area in `h3516.tr3`).

**Created/edited by:**

- `TrnToTr3.exe` / Landscape Editor `Tools -> Export terrain`.
- Condor_Tiles `Make Terrain (.TR3)`.
- Directly editable in Landscape Editor with height/flatten tools.
- **Must re-run `File -> Export terrain hash` after any change.**

---

### 3.3 `<LandscapeName>.tha` - terrain hash

**Purpose:** Anti-cheat / integrity file generated from all `.tr3` files. Condor will not load the landscape (and flight controls may be disabled) if this is missing or stale.

**Format:** plain ASCII text, CRLF line endings.

```text
000000 3057700
000001 338640
000002 1339961
...
3927   <hash>
```

Each line maps a patch coordinate (`CCRR`) to a numeric hash/area value.

**Created/edited by:** Landscape Editor -> `File -> Export terrain hash`.

---

### 3.4 `<LandscapeName>.apt` - airports

**Purpose:** Defines flyable airports in the landscape.

**Format:** binary, fixed-length records. Each airport is **72 bytes**.

| Offset (within record) | Field | Type | Notes |
|------------------------|-------|------|-------|
| 0 | name length | uint8 | Number of ASCII chars in the name. |
| 1-31 | name | char[31] | Null-padded ASCII airport name. |
| 32-35 | (unused) | float | Always `0.0` in Slovenia2. |
| 36-39 | latitude | float32 | Decimal degrees (WGS 84). |
| 40-43 | longitude | float32 | Decimal degrees. |
| 44-47 | elevation | float32 | Meters above sea level. |
| 48-51 | runway direction | int32 | Degrees (e.g. `134`, `111`, `273`). |
| 52-55 | runway length | int32 | Meters. |
| 56-59 | frequency / flags | int32 | Airport COM / misc data. |
| 60-67 | (unknown / padding) | - | Usually zero. |
| 68-71 | flatten radius / flags | float/int | Runway-flattening parameter and flags. |

**Slovenia2 example:** 8 airports (records are `576 / 72 = 8`), including `LESCE-BLED`, `CELJE`, `AJDOVSCINA`, `MURSKA SOBOTA`, `NOVO MESTO`, `POSTOJNA`, `PTUJ`, `SLOVENJ GRADEC`.

**Created/edited by:**

- Landscape Editor -> `View/Modify -> Airports -> Add/Edit`.
- Condor_Tiles `Airport Place` for fine runway adjustment.
- **Not hand-editable** because it is binary; generate it from the editor.


---

### 3.5 `<LandscapeName>.cup` - turnpoints / waypoints

**Purpose:** SeeYou `.cup` waypoint file used by the Condor flight planner and PDA.

**Format:** CSV text (ISO-8859 / Windows-1252), CRLF line endings.

```csv
name,code,country,lat,lon,elev,style,rwdir,rwlen,freq,desc
"AFRITZ CHURCH",AFRITZ C,,4643.650N,01347.983E,710.0m,1,,,,
"AJDNA",AJDNA,,4625.283N,01407.983E,995.0m,1,,,,
```

- `lat` / `lon`: degrees and decimal minutes with hemisphere (`DDMM.mmmH`).
- `elev`: meters with `m` suffix (or feet with `ft`).
- `style`: waypoint type. `1` = turnpoint/landmark; `2`/`4`/`5` = airfield types (standard SeeYou codes).
- `rwdir`, `rwlen`, `freq`: runway direction, length, frequency (used for airfield styles).

**Slovenia2 example:** 282 lines after header; all are style `1` turnpoints.

**Created/edited by:** Any text editor / spreadsheet, or SeeYou, or Condor_Tiles. Keep the header line.

---

### 3.6 `<LandscapeName>.obj` - 3D object placements

**Purpose:** Positions, scales and orientations of shared landscape objects (trees, buildings, power pylons, etc.) that reference `.c3d` models in `World/Objects/`.

**Format:** binary sequence of fixed 152-byte records.

```c
struct ObjRecord {
    float posX;          // relative to landscape easting
    float posY;          // relative to landscape northing
    float posZ;          // altitude above sea level / terrain offset
    float scale;         // uniform scale (1.0 = original size)
    float orientation;   // radians
    uint8 nameLen;       // length of model filename
    char  name[131];     // null-padded .c3d filename
};
```

Absolute UTM position is reconstructed as:

```text
Xabs = easting  - posX
Yabs = northing + posY
```

**Slovenia2 example:** `1139392 / 152 = 7496` objects. Typical record: `C1R.c3d pos=(180898, 93777, 488) scale=1.0 ori=4.712`.

**Created/edited by:**

- Landscape Editor -> `View/Modify -> Objects`.
- `condor_obj_file_tool.py` (community tool) can export to JSON and re-import.

---

### 3.7 `World/Objects/*.c3d` and `World/Textures/*.dds`

**Purpose:** Shared 3D models and their textures referenced from `.obj`.

**Format:** Proprietary Condor 3D model / DDS texture. Not documented here; treat as opaque binaries.

**Created/edited by:**

- `ObjectEditor.exe` for `.c3d` models.
- External image tools + NVIDIA texture tools / `nvdxt.exe` for `.dds`.

---

### 3.8 `Textures/tCCRR.dds` - ground tile textures

**Purpose:** Photo-realistic or generated ground texture for each tile.

**Format:** DDS texture. Common sizes: `4096x4096` or `8192x8192` per tile (Slovenia2 uses `8192x8192` tiles, ~5.6 MB each).

**Slovenia2 example:** 1120 tiles (`t0000.dds` ... `t3927.dds`) plus `empty.dds` fallback.

**Created/edited by:**

- Place source `XXYY.bmp` + water mask `aXXYY.bmp` in `Working/Terragen/Textures/`.
- Run `WaterAlpha` to merge and convert to `.dds`.
- Or import via Landscape Editor `Tools -> Import tile sized textures`, then `File -> Export textures to DDS` (requires `nvdxt.exe` in the CLT folder).

---

### 3.9 `ForestMaps/CCRR.for` - per-patch forest mask

**Purpose:** Defines where trees are rendered for each patch.

**Format:** binary, no header.

- `512 x 512` unsigned 8-bit values.
- File size: `262 144` bytes.
- Typical values:
  - `0` = no trees
  - `1` = coniferous / default forest
  - `2` = deciduous (depends on landscape setup)

**Slovenia2 example:** 1120 masks (`0000.for` ... `3927.for`).

**Created/edited by:**

- Generate from OSM / land-cover data with Condor_Tiles or `create_landscape.py`.
- Edit as `Working/ForestMaps/bXXYY.bmp` and `sXXYY.bmp` in the Landscape Editor.
- **Must re-run `File -> Export forest hash` after any change.**

---

### 3.10 `<LandscapeName>.fha` - forest hash

**Purpose:** Anti-cheat / integrity file generated from all `.for` files.

**Format:** plain ASCII text, CRLF line endings.

```text
000000 294909
000001 34117
...
```

**Created/edited by:** Landscape Editor -> `File -> Export forest hash`.

---

### 3.11 `<LandscapeName>.tdm` - thermal / albedo map

**Purpose:** Guides the Condor weather engine on thermal probability/strength.

**Format:** binary.

- **Header - 8 bytes:** `width` (int32), `height` (int32) matching `.trn` dimensions.
- **Data:** `width x height` unsigned 8-bit values (`0-255`).
  - `0` = no thermals (water, city, etc.)
  - Higher values = stronger/better thermals.

**Slovenia2 example:** `2560 x 1792` pixels, file size `8 + 2560*1792 = 4 587 528` bytes, value range `0-228`.

**Created/edited by:**

- Create/edit a 24-bit `Working/ThermalMap.bmp`.
- Landscape Editor -> `File -> Export thermal map` to produce `.tdm`.
- Community GIMP plugin can also read/write `.tdm` directly.


---

### 3.12 `<LandscapeName>.bmp` - PDA / flight-planner map

**Purpose:** 2D map shown in the Condor flight planner and PDA.

**Format:** Windows 32-bit BMP (`RGBA` or `XRGB`).

**Slovenia2 example:** `2560 x 1792 x 32 bit`, 18 350 136 bytes.

**Created/edited by:** Landscape Editor -> `File -> Export flightplanner map`. It is generated from the landscape overview.

---

### 3.13 `<LandscapeName>.ini` - landscape options

**Purpose:** Small options file.

**Format:** plain text, CRLF.

```ini
[General]
Version=1.00
RealtimeShading=1
```

- `Version` - shown in the server list / flight planner.
- `RealtimeShading` - `1` (default/on) or `0` to disable real-time terrain shading for photo textures.

**Created/edited by:** Any text editor.

---

### 3.14 `Images/*.jpg` - splash screens

**Purpose:** Loading-screen images shown when the landscape starts.

**Format:** JPEG.

**Slovenia2 example:** `Images/0.jpg` ... `Images/6.jpg`.

**Created/edited by:** Any image editor.

---

### 3.15 `Airports/` - airport scene files

**Purpose:** Custom 3D airports (runway/apron ground polygons and objects).

**Layout:**

```text
Airports/
├── <ICAO>G.c3d      # ground / apron model
├── <ICAO>O.c3d      # objects / buildings model
└── <ICAO>/          # textures for that airport
    └── *.dds
```

**Slovenia2 example:** `Airports/LJAJ/AjdovscinaG.c3d`, `Airports/LJAJ/AjdovscinaO.c3d`, `Airports/LJAJ/*.dds`.

**Created/edited by:**

- `ObjectEditor.exe` / `LandscapeEditor.exe` airport editor.
- Ground polygons are often built in Blender/3ds Max and exported to `.c3d`.

For the first flyable version of the North Macedonia / Skopje landscape you need at least:

| 3 | `<LandscapeName>.tha` | Landscape Editor -> `Export terrain hash` |
| 4 | `<LandscapeName>.apt` | Landscape Editor -> add airports |
| 5 | `<LandscapeName>.cup` | Text editor / SeeYou / Condor_Tiles |
| 6 | `<LandscapeName>.ini` | Text editor |
| 7 | `<LandscapeName>.bmp` | Landscape Editor -> `Export flightplanner map` |
| 8 | `<LandscapeName>.tdm` | Landscape Editor -> `Export thermal map` (can be blank/default initially) |
| 9 | `ForestMaps/CCRR.for` (initially empty) | Landscape Editor / Condor_Tiles |
| 10 | `<LandscapeName>.fha` | Landscape Editor -> `Export forest hash` |
| 11 | `Textures/tCCRR.dds` + `empty.dds` | WaterAlpha / Landscape Editor DDS export |
| 12 | `Images/*.jpg` | Image editor |
| 13 | `World/Objects/*.c3d` + `World/Textures/*.dds` | ObjectEditor (optional for Phase 1) |
| 14 | `Airports/` | ObjectEditor / Landscape Editor (optional for Phase 1) |

> **Note:** Even an empty forest map must be exported and hashed, otherwise Condor reports the landscape as incomplete and may disable flight controls.

## 5. Exact steps to create a new landscape from a `.trn` file in Landscape Editor

These are the steps used to go from a raw `.trn` heightmap to a flyable base landscape. They are distilled from the *Condor Landscape Guide* and community tutorials.

### 5.1 Prepare the folder structure

1. Create the landscape root folder, e.g.
   ```text
   C:/Condor2/Landscapes/MacedoniaSkopje/
   ```
2. Inside it create the subfolders:
   ```text
   HeightMaps/
   Textures/
   ForestMaps/
   Images/
   World/Objects/
   World/Textures/
   Airports/
   Working/
   Working/Terragen/Textures/
   Working/Terragen/ForestMaps/
   ```
3. Copy your generated `.trn` file into the root and name it exactly like the folder, e.g. `MacedoniaSkopje.trn`.

### 5.2 Open the landscape

4. Start `LandscapeEditor.exe` from `tools/CLT2.7/`.
5. Select the new landscape from the **Landscape** drop-down in the top-right.
   - If it is not listed, make sure the folder and `.trn` file names match and are under `C:/Condor2/Landscapes/`.
6. You should see the terrain grid. Use the **FINGER** tool to pan; left-click zooms in, middle/right-click zooms out.

### 5.3 Add at least one airport

7. In the **View/Modify** panel check **Airports**.
8. Right-click in the blank area below and choose **Add**.
9. Fill in:
   - Name
   - Latitude / Longitude (decimal degrees, runway center)
   - Elevation (meters)
   - Runway direction (degrees true)
   - Runway length (meters)
   - COM frequency / other fields as needed
10. Click **OK**. The airport marker appears on the map.
11. (Optional) Select **Height Map**, zoom to the runway, click the **FLATTEN** icon, set the runway elevation as altitude, a radius no larger than the runway, and an edge slope of about `1:4` to smooth the runway area.

### 5.4 Add turnpoints (optional)

12. Create `<LandscapeName>.cup` with at least a header line and one dummy waypoint, or copy/rename the Slovenia2 `.cup` and edit it. Place it in the landscape root.

### 5.5 Export the base files

13. `File -> Export flightplanner map` -> creates `<LandscapeName>.bmp`.
14. `File -> Export forest map` -> creates the initial forest map / `.for` files (may be empty).
15. `File -> Export thermal map` -> creates `<LandscapeName>.tdm`.
    - If the editor complains that no thermal map is loaded, place a 24-bit `Working/ThermalMap.bmp` (same pixel size as `.trn`) and repeat.
16. `File -> Export textures to DDS` -> creates `Textures/tCCRR.dds`.
    - Requires `nvdxt.exe` (NVIDIA legacy texture tools) in the same folder as `LandscapeEditor.exe`.
    - Answer **No** when asked to export only non-existing files to rebuild everything.
17. `File -> Export terrain hash` -> creates `<LandscapeName>.tha`.
18. `File -> Export forest hash` -> creates `<LandscapeName>.fha`.
    - **Do not skip steps 17 and 18** - they are required for Condor to load the landscape and to avoid the "no elevator control" anti-cheat issue.
19. `File -> Save landscape`.

### 5.6 Test

20. Close Landscape Editor.
21. Start Condor 2 -> Free Flight -> select the new landscape.
22. Create a simple task around the airport to verify the terrain loads and controls work.

## 6. Key tools and their roles

| Tool | Path | Purpose |
|------|------|---------|
| `LandscapeEditor.exe` | `tools/CLT2.7/` | Main editor: import terrain, place airports/objects, export all base files and hashes. |
| `RawToTrn.exe` | `tools/CLT2.7/` | Convert raw/BIL/GeoTIFF heightmaps to `.trn`. |
| `TrnToTr3.exe` | `tools/CLT2.7/` | Convert `.trn` to patch `.tr3` files. |
| `ObjectEditor.exe` | `tools/CLT2.7/` | Create/edit `.c3d` object models. |
| `nvdxt.exe` | copy into `tools/CLT2.7/` | NVIDIA DDS compressor used by `Export textures to DDS`. |
| `WaterAlpha` | community tool | Merge RGB texture BMPs with water-alpha BMPs into final `.dds`. |
| `Condor_Tiles.exe` / `create_landscape.py` | community | Automate DEM download, `.trn`/`.tr3` generation, texture/forest/thermal tile creation. |
